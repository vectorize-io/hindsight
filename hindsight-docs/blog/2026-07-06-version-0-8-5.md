Hindsight 0.8.5 brings **startup robustness improvements**, **distributed bank statistics caching** for faster dashboard loads, **structured recall filtering** for finer precision control, and several **critical LLM configuration fixes** that ensure your provider settings are honored correctly. **If you're using batch fact extraction, per-scope LLM configs, or custom headers with LiteLLM providers, upgrade now.**

- [**Distributed Stats Caching**](#distributed-stats-caching): Faster bank statistics with manual refresh.
- [**Structured Recall Filtering**](#structured-recall-filtering): Two-level min_scores for better precision control.

## Startup Improvements

The startup process is now more robust with comprehensive zombie process cleanup and improved health monitoring. Stale worker processes, hung API instances, and orphaned PostgreSQL connections are detected and cleaned up automatically, reducing manual intervention after unexpected shutdowns. This is especially useful in development environments with frequent restarts.

## Distributed Stats Caching

Bank statistics are now cached in a distributed manner, dramatically speeding up dashboard loads when viewing multiple banks. A new `?refresh=true` query parameter forces a fresh calculation when you need up-to-date numbers. The change includes a performance test suite to ensure stats remain fast even with millions of memories.

## Structured Recall Filtering

Recall now supports **structured per-stage scores** and **two-level min_scores filtering**, giving you finer control over which memories make it through each stage of the retrieval pipeline. Set different thresholds for semantic search, keyword matching, and reranking to tune precision vs. recall for your use case. This is especially valuable when working with noisy data or when you need to enforce strict relevance requirements.

## Critical LLM Fixes

Several fixes ensure LLM provider configuration is honored correctly across all code paths:

- **Batch extraction temperature.** The configured LLM temperature is now properly applied in batch fact-extraction jobs. Previously, the batch path ignored your temperature setting and used the model's default.
  
- **Custom headers with LiteLLM.** `default_headers` are now correctly wired into all LiteLLM-backed providers (Anthropic, OpenAI via LiteLLM, Fireworks, etc.). If you set custom headers (for example, for analytics or routing), they'll now reach the provider.

- **Per-scope timeouts and retries.** Per-scope LLM timeout and retry policies are now propagated to the provider correctly. If you configured different timeouts for retain vs. reflect, those settings are now respected.

- **Per-operation temperature.** Per-operation temperature (retain, reflect, consolidation) is now fully configurable. Set `HINDSIGHT_API_RETAIN_LLM_TEMPERATURE` independently from `HINDSIGHT_API_REFLECT_LLM_TEMPERATURE` and they'll work as expected.

Other notable improvements:

- **Vector extension bootstrap.** Migration bootstrap now respects the configured vector extension (pgvector vs. pgvectorscale) instead of defaulting to pgvector.
- **Conversation chunking.** Retain now merges JSON arrays in append mode, preserving conversation-aware chunking for chat logs and structured data.
- **Consolidation vectors.** Search vectors are now populated on observation INSERT/UPDATE during consolidation, ensuring newly consolidated memories are immediately searchable.
- **Reflect output cleanup.** Reflect now unwraps JSON answer envelopes automatically, producing cleaner final outputs without manual post-processing.
- **Control plane uploads.** The control plane now shows pending uploaded documents from server operations, improving visibility into async ingestion jobs.
